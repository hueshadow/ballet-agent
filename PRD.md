## 1. 产品背景

### 1.1 市场现状
- 中国成人芭蕾市场近 5 年快速增长，一线及新一线城市涌现大量中小型工作室
- 几乎每家机构都有自建约课小程序或第三方 SaaS（美团丽人、小鹅通定制、有赞教育等）
- 学员被要求在每家机构单独注册、单独购课、单独预约

### 1.2 用户痛点
| # | 痛点 | 描述 |
|---|---|---|
| P1 | 课时分散 | 在 A 机构上 3 节，在 B 机构上 5 节，没有任何一处能看到"我今年一共上了几节课" |
| P2 | 成长不可见 | 无法基于总课时对自己的水平做评估；看不见长期进步 |
| P3 | 课包绑定 | 大课包便宜但量大上不完、小课包贵；学员只想上"某位老师"的单课，但机构不卖 |
| P4 | 记录易丢 | 机构小程序经常变更/下架/更换 SaaS，历史课时被清空 |
| P5 | 分享麻烦 | 上完课想发小红书，每次都要自己 P 图、写文案、做日签 |

### 1.3 产品愿景（长期）
做一个接入多家芭蕾机构、支持"通用课时积分钱包"的微信小程序，让成人芭蕾爱好者：
- 一个账号管理所有机构的课时
- 用通用积分跨机构按需购课
- 拥有可量化的成长档案和水平评估
- 轻松生成和分享上课记录

### 1.4 MVP 定位（本期目标）
**不依赖任何机构接入**，先做"个人芭蕾成长档案"工具，目标用户只有产品负责人本人。
- 先用自己的数据跑通产品闭环
- 数据模型从第一天起为未来的"多用户 + 多机构 + 积分钱包"留位置
- 建立第一批真实种子数据（你过往的上课记录），为 V2 上线后的冷启动做准备

## 2. 用户与场景

### 2.1 用户画像（MVP 仅 1 个）
- **姓名**：产品负责人本人
- **城市**：苏州
- **身份**：成人业余芭蕾爱好者
- **舞龄**：5 年以上
- **常去机构**：固定 3-4 家
- **常上课型**：基训、把杆、中间、编舞
- **使用动机**：记录、追踪进步、沉淀成长档案、方便分享到小红书
- **技术水平**：会用微信小程序，不写代码
- **身份双重**：本项目产品 + 设计 + 开发一人担，同时是唯一用户

### 2.2 核心使用场景

**场景 A：下课后 3 分钟内快速打卡**
> 上完一节课，在机构门口拉伸休息时，打开小程序，选机构/老师/课型/时长，上传 1-2 张课后合影，写两句感受，保存。顺手生成一张 AI 日签，保存相册，打开小红书发布。

**场景 B：周末批量补录上周课时**
> 周日晚上花 10 分钟把本周上的所有课一次性补录进去，看一眼本周总课时和月度进度条。

**场景 C：第一次使用，补录过往半年的历史数据**
> 第一次打开小程序，花 30-60 分钟把最近半年的上课记录手动逐条补录（凭记忆 + 机构小程序历史订单截图对照）。

**场景 D：月度动作打卡**
> 每个月月初，在家里或教室角落录一段 plié（30 秒内），上传到动作库。随时能打开动作详情页，把本次视频和半年前的视频并排播放对比。

**场景 E：翻看成长档案**
> 跨年/生日/节日时，翻开看板，看看今年总共上了多少节课、哪位老师的课最多、哪一个月最勤奋。

### 3.2 M1 · 用户档案

**功能**：记录个人基础信息，作为成长档案的主体。

**字段**：
- 头像、昵称（从微信资料获取）
- 舞龄（月数，可手填）
- 个人简介（选填）
- 学习目标（例：完成考级 / 参加汇演 / 上足尖）

**交互**：首次进入小程序引导填写；之后可随时在"我的"页面修改。

### 3.3 M2 · 上课记录 🎯 核心

**3.3.1 新增上课记录**
- 日期（默认今天，可改）
- 机构（从列表选 / 快速新建）
- 老师（按机构联动筛选 / 快速新建）
- 课程类型（基训 / 把杆 / 中间 / 跳跃 / 足尖 / 编舞 / 汇演排练 / 其他，**支持用户自定义新增**并存入个人词典）
- 课程级别（初级 / 中级 / 高级）
- 时长（默认 60 分钟，可改）
- 课后照片（**不限张数**，前端按需压缩）
- 课堂笔记 / 自我感受（多行文本，**支持语音输入转文字**，走微信小程序 recognizer API）
- 支付方式（机构课包 / 单次付费 / 赠送 / 其他） · *V2 预留字段*
- **[新增] 身体状态评分**（1-5 星，记录今日身体状态/疲劳度，为未来伤病追踪埋点）
- **[新增] 老师带课评分**（1-5 星，记录该节课老师带课的体感）
- **[新增] 主题/关键词标签**（多个 chip，如"跳跃"、"起增"、"Adagio"、"足尖准备"；支持自由输入+历史复用）
- **[新增] 课后愉悦度 emoji**（单选，约 8 个预设 emoji：🥰😊😌😐😖😭💪✨）
- 记录来源（系统自动标记：今日打卡 / 历史补录 / CSV 导入）

**3.3.2 列表页**
- 时间倒序展示
- 筛选：机构、老师、月份、课程类型、主题标签、愉悦度
- 搜索：笔记关键字 / 标签关键字
- 每条展示：日期、机构 logo、老师名、课型、时长、首张照片缩略、愉悦度 emoji

**3.3.3 详情页**
- 完整信息展示
- 照片轮播、点击放大
- 编辑 / 删除
- "一键生成日签"按钮（跳转 M5）
- 快捷复制："以这条为模板新建一条"（访谈新增，用于减轻手录压力）

### 3.4 M3 · 课时统计看板 📊

**首页卡片**：
- 本月课时（节数 / 小时数）
- 累计课时（节数 / 小时数）
- 本周进度条（目标 vs 实际，**默认周目标 3-4 节**，可在"我的"设置修改）
- 最近一节课距今天数

**统计页图表**（图表库：**ec-canvas**，即官方微信版 ECharts）：
- 近 12 个月课时折线图
- 机构分布饼图（"苏州 A 机构 60% / B 机构 40%"）
- 老师分布条形图 Top 10
- 课型分布饼图
- 成长里程碑：第 10 / 50 / 100 / 200 / 500 / 1000 节课 的达成时间（用户舞龄 5 年+，进度条从 200 节开始也合理）

### 3.5 M4 · 动作视频打卡库 🎥 差异化核心

**预置动作清单（MVP 固定 10 个）**：
1. Plié（蹲）
2. Tendu（擦地）
3. Dégagé（小踢腿）
4. Rond de jambe（划圈）
5. Frappé（弹动）
6. Fondu（融化）
7. Développé（控腿）
8. Grand battement（大踢腿）
9. Relevé（上升）
10. Pirouette 单圈（旋转）

每个动作内置：中文名、法文名、发音提示、简要说明文案、**AI 预生成示范图（通过 modern-frontend-design skill 和 Gemini 一次性生成 10 张统一风格的插画，存入云存储）**。

**M4.1 动作库首页**
- 10 张动作卡片网格（2×5 或 3 列滚动）
- 每张卡片显示：动作名、已打卡次数、最近一次距今天数、进度徽章

**M4.2 单个动作详情页**
- 基础说明
- 时间线：按日期倒序展示历次打卡视频缩略图
- "对比"按钮：选择任意两次打卡，**MVP 仅做并排双视频独立播放**（各自拖动/各自暂停），同步播放 + 变速 0.5x/1x 放到 V1.1
- "新打卡"按钮

**M4.3 打卡页**
- 调用 `wx.chooseVideo` 录制/选取（**不限时长，但界面提示"建议 30 秒内"**；前端压缩）
- 自动生成封面缩略图
- 自评 1-5 星
- 笔记（今天我的感受）
- 保存

### 3.6 M5 · AI 日签 🎨

**功能流**：
1. 在上课记录详情页点"生成日签"
2. 系统基于课时信息（日期、机构、老师、课型、笔记、主题标签、愉悦度 emoji）组合 prompt
3. **云函数调用 Gemini（Imagen / Nano Banana 系列）生图**
   - Gemini 国内不可直达，云函数 → **用户自有的 Cloudflare Workers / 海外 VPS 代理** → Gemini
   - 代理地址与 API Key 放云函数环境变量：`GEMINI_PROXY_URL`、`GEMINI_API_KEY`
   - 不下发到小程序端，不硬编码
4. 返回一张 1080×1920 竖版图（芭蕾主题插画 + 当日文案 + 日期水印）
5. 用户预览，可"再生成一张"
6. 满意后保存到云端日签库
7. 长按保存到手机相册（**MVP 分享只做保存相册，其他手动**，小红书无公开发布 API）

**兜底机制**：
AI / 代理调用失败时，用小程序原生 Canvas 2D 合成一张静态海报（预设 5-10 套模板），保证"一定能出图"。Canvas 模板走与小程序主界面一致的极简黑白风格（见第 11 章设计系统）。

**文案库**：
MVP 准备 50-100 句芭蕾相关短句（诗句 / 名言 / 课堂用语 / 心情短句），随机抽取叠加在日签上。

### 3.7 M6 · 机构 / 老师管理

**MVP 版本**：新增上课记录时可"快速新建"；同时在"我的"入口提供统一管理页（列表 / 新建 / 编辑 / 删除），不做老师独立详情页。
- 机构字段：名称、城市、地址（选填）、logo（选填）
- 老师字段：名称、所属机构（可多选）、简介（选填）、头像（选填）

**V2 会演进为**：老师独立详情页（上课统计、照片集、平均评分）、机构后台同步、机构入驻管理。

### 3.8 M7 · 个人课表（新增 P0）

**功能**：作为"上课提醒"的数据源，用户手动维护自己的未来课表。

**3.8.1 课表字段**
- 日期 + 开始时间 + 预计时长
- 机构（联动 M6）
- 老师（联动 M6）
- 课程类型（可选，联动 M2 课型词典）
- **重复规则**（每周 / 隔周 / 不重复）
- 提醒提前量（默认课前 30 分钟）

**3.8.2 课表首页**
- 日历视图（月 + 周切换）
- 即将到来的 7 天课程卡片
- 从课表条目一键 →"生成新上课记录"（预填机构/老师/课型/时长）

**3.8.3 提醒机制**
- 课前提醒：基于用户订阅消息 + 云函数定时任务（或直接使用小程序模板消息）
- 课后补录提醒：课程预计结束时间 + 2 小时后提醒"补一条上课记录"
- 提醒通过微信订阅消息下发，不强制，用户可主动关闭

**3.8.4 边界**
- 不做机构排课同步（V2 机构接入后才有意义）
- 不做家人/好友课表共享

### 3.9 M8 · CSV 批量导入历史课（新增 P0）

**背景**：用户舞龄 5 年+、希望补录"尽可能多"的历史课。如果全部手录，可能上千条，微信小程序手机端无法承受。

**功能流**：
1. "我的"页 → 导入数据 → 下载 CSV 模板
2. 用户在电脑端 Excel 或 Numbers 填写（字段与 M2 记录对齐，机构/老师用名称即可，不存在则新建）
3. 回到小程序 → 选择文件 →"上传 CSV"
4. 小程序读取 → 预览（显示前 N 条 + 总数 + 警告/错误）
5. 用户确认 → 云函数批量写入（分批 + 进度条）
6. 成功后跳转到记录列表

**校验规则**：
- 必填：日期、机构、老师、课型、时长
- 日期格式：YYYY-MM-DD
- 机构/老师若不存在，自动新建并标记"待补完信息"
- 去重：同一天 + 同机构 + 同老师 + 同时长 视为疑似重复，预览时标黄提示
- 历史记录的 source 自动标记为 `csv_import`
- 一批最多导入 1000 条，超出强制拆分

**边界**：
- CSV 不含照片字段；照片在导入成功后用户可按条补充
- 不支持 .xlsx（只接 .csv UTF-8）

### 3.10 M9 · 一键导出 JSON（新增 P0）

**功能**：
- "我的"页 → 数据 → 一键导出
- 云函数遍历当前用户名下的所有集合（users / studios / teachers / lessons / moveCheckins / dailySigns / schedules）
- 打包成一个 JSON 文件（照片/视频以 fileID 引用，不嵌入二进制）
- 通过小程序 `wx.saveFileToDisk` 或复制链接方式交给用户保管

**Why**：用户将补录上千条珍贵历史数据，需要应急数据备份能力。完整的导出/导入在 V1.1 做，MVP 只做导出（只读）。

**边界**：
- MVP 不做导入恢复（导入只支持 M8 的 CSV 格式）
- 不做增量导出；每次全量

## 4. 非功能需求

### 4.1 性能
- 首页打开 < 1.5 秒
- 上课记录列表（100 条内）滚动流畅
- 视频上传 30 秒内完成（5MB 以下）

### 4.2 可用性
- 弱网环境下所有上传操作必须有明确的 loading / 失败重试
- 所有列表页有空态插画 + 引导文案
- 所有表单有字段级校验

### 4.3 隐私与数据
- **从第一天起严格按 `_openid` 过滤**：所有云函数查询、数据库 where 条件、文件访问都必须加 `_openid = auth.openid` 条件，V2 多用户零成本迁移
- AI 生图 API Key 和代理地址只在云函数环境变量中使用，不下发到小程序端
- 视频/照片默认仅当前用户可见
- "一键导出 JSON" 作为 MVP 兜底能力，V1.1 扩展至完整的导出/导入/年度报告

### 4.4 成本
- 云开发套餐：MVP 阶段使用免费额度（数据库 2GB、存储 5GB、CDN 流量 5GB/月）
- AI 生图成本：单张预估 0.05-0.3 元，MVP 阶段个人用量可控

## 6. 技术方案（概要）

| 层 | 选型 | 理由 |
|---|---|---|
| 前端 | 微信原生小程序 | 单端部署，不引入多端抽象成本 |
| 后端 | 微信云开发 CloudBase | 免运维、免备案、官方背书 |
| 数据库 | 云开发文档型数据库 | 与 CloudBase 原生集成 |
| 文件存储 | 云开发文件存储 | 直接返回 fileID 给前端渲染 |
| AI 生图 | **Gemini（Imagen / Nano Banana）**，云函数 → 用户自有 Cloudflare Workers / 海外 VPS 代理 | 画质最优；用户已有代理资源 |
| 图表 | **ec-canvas**（官方微信版 ECharts） | 社区活跃、文档齐全 |
| 海报合成 | Canvas 2D | 日签模板兜底 |
| 语音输入 | 微信小程序 `wx.getRecorderManager` + 插件识别 | M2 笔记语音转文字 |
| 设计资产 | **`modern-frontend-design` skill 统一产出** | 保持极简黑白风格一致性，避免 AI 默认 SaaS 外观 |

**导航结构**：5 底部 tab
1. 首页（M3 看板）
2. 记录（M2 列表）
3. 中间 + 号（快速打卡入口，点击弹出新建上课记录表单）
4. 动作库（M4）
5. 我的（M1 档案 + 目标设置 + M6 机构老师管理 + 课型管理 + M8/M9 数据导入导出 + 关于/版本/隐私声明）

**开发阶段拆分**：脚手架 & 云开发初始化 → 数据底座 & 鉴权 & `_openid` 隔离 → M1/M6 档案与机构老师管理 → M2 登记/列表/详情（含 M8 CSV 导入） → M3 统计看板 → M7 课表与提醒 → M4 动作打卡 → M5 AI 日签 → M9 JSON 导出 → 打磨体验版上线。

## 10. 设计系统与风格基线

### 10.1 视觉基调
- **风格**：极简黑白（Apple / Linear 风），大量留白，克制使用色彩
- **禁止**：Material Design 默认视觉、Ant Design 默认视觉、彩色渐变背景、大圆角 + 大阴影的"soft UI"、满屏 emoji
- **允许**：1-2 个点缀色（用于目标达成、日签预览、里程碑徽章等强调场景），其他全部走黑白灰 + 字体层级

### 10.2 资产产出方式
所有设计资产（logo、色板、字号、间距 token、组件规范、空态插画、日签 Canvas 模板、动作库示范图）**必须通过 `modern-frontend-design` skill 统一产出**，按以下节奏：

1. 写完 OpenSpec bootstrap change proposal 后，第一件事用 skill 产出"设计系统基线"文档（token + 关键页面 HTML mockup）
2. 每个 P0 模块的 change proposal 实现阶段，先用 skill 出该页的 HTML mockup 给用户审阅
3. 通过后再进小程序页面实现，严格对齐 mockup 的间距/字号/组件

### 10.3 内容主角
用户拍摄的课后照片、动作打卡视频、日签图是视觉主角，界面本身要"让位"。所有容器、按钮、字重的设计都遵循"不与内容争焦点"原则。

## 11. 开发与发布策略

### 11.1 流程：严格走 OpenSpec
- 每个 P0 模块开发前单独写 OpenSpec change proposal
- proposal 内必须包含**文字描述的 UI 交互脚本**（页面流程、状态、边界、错误处理）
- 共享基建（脚手架 / 云开发初始化 / 基础 schema / 鉴权 / `_openid` 中间件）合并为 1 个 "bootstrap" change proposal
- proposal → 用户审阅 → 通过 → 进 implementation → 完成 → archive

### 11.2 发布：不上架
- **MVP 只发布小程序后台的"体验版"**，产品负责人本人用体验版链接 / 扫码进入
- 跳过提交审核、跳过备案、跳过模板消息（课表提醒用体验版可用的订阅消息）、跳过内容安全检测、跳过个人主体验证
- V2 启动商业化前再补齐发布/合规

### 11.3 环境前置
开发启动前需要用户完成：
- 申请微信小程序 AppID（个人主体即可）
- 创建云开发环境（体验免费额度）
- 准备 Gemini API Key 并配置好 Cloudflare Workers / VPS 代理（`GEMINI_PROXY_URL` + `GEMINI_API_KEY`）

这些准备步骤随 bootstrap change proposal 一起交付给用户执行。
